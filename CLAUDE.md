# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run build` — typecheck (`tsc`, strict) + production build. This is the verification step after every change; there are no tests or linters.
- `npm start` / `npm run dev` — Vite dev server on port 3000 (the app's registered App URL).
- `npm run typecheck` — `tsc` only.

Git: commit messages must NOT contain `Co-Authored-By` or any AI-attribution trailer (user rule). **Keep messages short** (user rule) — a subject line, and a body only when something genuinely won't survive without it. Ignore the long multi-paragraph bodies in `git log`; they are not the convention. Rationale, rejected alternatives and product decisions go in [docs/DECISIONS.md](docs/DECISIONS.md), which exists for exactly that. **master is production** — every push auto-deploys to GitHub Pages via `.github/workflows/deploy.yml`; develop on the `preview` branch and merge to master only to ship.

To see changes in Miro: the app must be registered once (README has the steps; App URL `http://localhost:3000`, scopes `boards:read` + `boards:write`). After panel changes, close and reopen the panel; after changes to `src/index.ts` or anything it pulls in (the features/adapters it wires), the whole **board tab must be refreshed** — the headless page only reloads with it. One-time listener registrations are guarded by `window` flags and survive HMR. `Miro creds.txt` in the root holds credentials; never read, modify, or commit it.

## What this is

A Miro Web SDK v2 app for event modeling: a palette panel that places typed building blocks (sticky notes, screens, automations, slices), pattern stamps, swimlanes, chapter markers, and Given/When/Then specification frames with linked typed copies. Blocks can carry named, typed **fields** (rendered in the sticky's own text, or in an attached box for screens/automations); a background **completeness** check reddens every arrow into a block whose incoming blocks don't supply all of its required fields between them, captioning each with the shortfall. A sticky's fill color already denotes its block type (so a plain colored sticky is recognized without any conversion step); plain **frames** drawn with Miro's own tools can be **converted** into slices or specs. It also has an AI **generator** that drafts a whole model — blocks, slices, links, fields, and specs — from pasted prose (Claude, via the user's own API key).

Two entry pages, both declared as Vite build inputs:

- `index.html` → `src/index.ts` — the **headless board script**. Loaded invisibly for the whole board session. Owns everything that must work with the panel closed: the `icon:click` handler, the `selection:update` flow (on-board + buttons, fast frame-size watcher for spec reflow and slice-button re-dock), and the background passes (`specHousekeeping`: copy sync, dead-spec/slice cleanup, fallback reflow/re-dock, button healing; `fieldsHousekeeping`: box re-render/re-dock; `completenessHousekeeping`: pool the required fields of everything pointing into a target and, when they don't cover the target's required fields (matched by name; type is not compared), redden every arrow into it and caption it with the shortfall; even when the pool does cover it, an arrow whose own source supplies none of what the target needs is still reddened on its own, tracked in the `em-flags` registry). **They are activity-driven, not polled on a fixed clock** (`domain/pacing`): `selection:update` is free, so it marks activity, and a completeness pass runs ~2s after activity settles (debounced, so a drag collapses to one pass). The heavier passes ride the same trigger, throttled to at most once per 30s. A 120s interval is a *safety net only* — for a change made with no local activity anywhere (a REST/bot edit) — and runs completeness alone. All of them stand down under `isUnderRateLimit()`. The fixed 4s poll this replaces cost the same on an untouched board as on one being edited, and exceeded the hourly credit budget by itself (see DECISIONS.md).
- `app.html` → `src/app.tsx` — the React 19 panel (Mirotone CSS from CDN). Owns only UI and the `drop` handler (drop targets are panel DOM elements).

## Code organization — ports & adapters, everything a component

The code is a **hexagonal (ports-and-adapters) architecture** so the
event-modeling logic can be lifted onto a non-Miro canvas: Miro is one
swappable adapter, never imported by domain or feature code. Layers, in
dependency order (each may import only from layers below it):

- `src/domain/` — **pure** event-modeling logic, zero platform deps. `vocabulary` (block types, colors, labels), `fields` (field types + the sticky-text/attached-box render + parse round-trip), `spec` (zone geometry + layout math), `slice` (slice geometry), `viewport` (expand-never-zoom math + `panTo`, the navigate-to counterpart: recenter at the current zoom, grow only to fit), `completeness` (the data-flow rule: which arrows are under-supplied), `fieldTransfer` (what a field looks like once copied across an arrow — alias and `!` stripped — and the name-keyed merge), `plan` (the `ModelPlan` the generator builds + `normalizePlan` trust boundary + slice/column layout math), `records` (registry/link/field/flag types + `normalizeRecords`), `pacing` (how often the background passes run — the active/idle policy), `credits` (the credit-budget arithmetic: per-second charge buckets + sliding-window sums), `diagnostics` (the failure-log entry shape), `meta` (element tag union), `naming` (`replaceFirstLine` — rewrite an element's name line, keep the body verbatim so a rename never round-trips a note's prose through the field parser). This is the portable core.
- `src/ports/` — the abstractions the use-cases speak to: `canvas` (the `Canvas` interface + the `CanvasElement` snapshot model and `ElementPatch`), `store`, `notifier`, `viewport`, `runtime`, `diagnostics` (the failure log behind the Console tab), `credits` (the credit meter behind it — a gauge, never a control), `planner` (text → `ModelPlan`, plus its own API-key/model config), plus `errors` (`HostUnavailableError` — the one distinction every port contract needs: could the host not be reached, or did it answer?). Ports import only domain types.
- `src/services.ts` — the service locator: `configureServices()` (called once per page at startup) + `services()`. The single seam where use-cases obtain ports; the domain never touches it.
- `src/features/` — the **use-cases** (one module per feature: `stickies`, `automation`, `screens`, `slices`, `createBlock`, `connectors`, `patterns` (eight pattern stamps; anchors on a single selected matching block), `swimlane`, `chapter`, `convert` (adopt plain frames as slices/specs; stickies need no conversion — their color already denotes the block type, and patterns/fields/completeness all read that color), `adoptImages` (the Screen tile's contextual click: adopt the selection's plain images as screens — tag + grouped title), `completeness`, `arrow` (the selected-arrow toolset: resolve both endpoints fresh, copy/replace fields across the link through the fields edit path, pan the viewport to either end), `naming` (the Properties tab's name editor: resolve the selection's one named element — sticky first line, screen/automation grouped title text, slice/spec/frame title, swimlane shape text, chapter caption — and rename it with fresh reads at commit time; no registry, the board display is the store), `registerDrop`, `generate` (build a whole model from a `ModelPlan`; the build is interruptible and resumable — progress is checkpointed to the `em-gen` board-app-data key after each block/link/spec, so Pause or a reload can resume from where it stopped), `generateCheckpoint`, `plannerSettings`, plus `helpers`/`assets`), `src/features/fields/` (`model` = domain re-export + store-backed reads + the text-vs-box display policy, `edit` = serialized registry upsert, `sync` = read board edits back into the registry, `render` = write the sticky text / attached box, `housekeeping` = poll-driven box healing & cleanup), and `src/features/specs/` (`model` = domain re-export + store-backed reads, `create`, `copies`, `reflow`, `selection`, `housekeeping`). Features read ports via `services()`; they never reference `miro` or `adapters/`.
- `src/adapters/miro/` — the **only** place the `miro` global appears. `MiroCanvas` (keeps a live-handle cache so a batched `apply` is one round-trip per item), `MiroStore`, `MiroNotifier`, `MiroViewport`, `MiroRuntime`, `meta` (`META_KEY`), `rateLimit` (the single gate every SDK call passes through — pacing, retry, cooldown, and the credit charge), `weights` (what each call costs, and the flagged inferences where the docs don't say); `index.ts` exports `createMiroServices(credits)`, which takes the meter and points `rateLimit` at it.
- `src/adapters/browser/` — host capabilities that are the browser's, not Miro's: `diagnostics` (the log — a per-page ring buffer, cross-frame over a **BroadcastChannel** since the board script and the panel are separate iframes, with replay so a panel opened later still sees earlier entries; optional `localStorage` persistence, **off by default**, written by the board page alone), `credits` (the credit meter — the same channel/replay/single-writer shape, but with rings kept **per page load** rather than merged, since credits carry no id to deduplicate by and a page's own re-broadcast would otherwise count twice; persistence is **unconditional** here, because an hourly figure that zeroed on refresh would be blank exactly when it's wanted) and `globalErrors` (the `window.onerror` / `unhandledrejection` net). No API credits are spent on any of it — which matters, since the failure they usually report is the credit budget running out. `createDiagnostics(source)` and `createCreditMeter(source)` are wired by both entry points, tagged `'board'` or `'panel'`.
- `src/adapters/anthropic/` — the **only** place the `@anthropic-ai/sdk` appears. `AnthropicPlanner` (calls Claude with a JSON-Schema-constrained structured output, then `normalizePlan`s it), the model catalog, the system prompt + schema, and `localStorage`-backed settings (the API key is per-browser, never board app data — `getSettings`/`setSettings` throw rather than pretend, so `GenerateSection` reads them **once** into a `SettingsLoad` and renders the failure instead of an empty key field); `index.ts` exports `createAnthropicPlanner()`. Only the panel page wires it.
- `src/panel/` — React components (`Panel` + `PanelTabs` (four tabs: **Build** = manual palette, **Properties** = the per-selection name + field editor, **Generate** = AI, **Console** = the failure log + the `CreditsMeter`) + one component per section (`BuildingBlocksSection`, `PatternsSection`, `ConvertSection`, `PropertiesSection` (the `NameEditor` on top, then an accordion of `FieldRow`s — each field a collapsed board-notation line, one open editor at a time; hands the tab to `ArrowSection` when the selection is exactly one attached connector, and shows the name editor alone for field-less named elements — slices, specs, swimlanes, chapters, notes), `ArrowSection` (the arrow toolset: copy/replace fields across the link, navigate to either end), `GenerateSection`, `ConsoleSection`), `NameEditor`, `Swatch`, `Dot`, `useBusyGuard`, `useDragReorder`, `useAutoPropertiesTab`, and `useSelection` (the selection-tracking hook the Properties/Convert sections share)), each with co-located plain CSS. Panel calls features, never ports/adapters directly.
- `src/index.ts` / `src/app.tsx` — composition roots: build `createMiroServices()`, `configureServices()`, then wire runtime events / mount the panel. `app.tsx` also adds `createAnthropicPlanner()` to the bundle (`Services.planner` is optional and panel-only, so the board script stays free of the AI SDK). Swapping the adapter set here is all it takes to host the tool elsewhere.

**Rule (user-mandated): every new piece of functionality is its own module/component, with its styles in a co-located plain `.css` file imported by the component.** (Plain CSS, not CSS Modules — class names are global because of the dynamic `bg-<type>` classes and the Mirotone/`miro-draggable` interplay.) Never grow an existing file into a god-file. `Panel.css` holds the shared section primitives (`.section`, `.footnote`, `.w-full`); `src/style.css` keeps only design tokens and the `bg-*` vocabulary colors.

**To extend:** put platform-free logic in `domain/`, the workflow in `features/` (talking only to ports), and any new SDK capability behind a port method implemented in `adapters/miro/`. Never call `miro.*` outside the adapter.

## Failures: never fabricate an answer (user-mandated)

**A failure is reported, never turned into a plausible-looking value.** This is the rule the codebase most expensively learned: every adapter used to `catch` its SDK errors and return `null` / `[]` / silence, so a rate-limited board was indistinguishable from an empty one — the Fields tab rendered "Select a command, event…" over an API failure and latched there until the board was refreshed (full story in [docs/DECISIONS.md](docs/DECISIONS.md)). Three rules follow, in order:

1. **Adapters propagate.** They never report and never catch. `getMeta` returning `null` means the board *answered* and the element carries no tag; a read that fails throws. Same for `store.read`/`write` (a swallowed write claimed to have persisted, which is how field records vanished against the app-data budget) and for the planner's `localStorage` settings (a swallowed read claimed the user had no API key). A **genuine absence is still a value**: no stored settings returns the defaults, no metadata returns `null` — only the *failure to look* throws.
2. **Only a supervisor catches** — the boundary around a background loop, which must not let one bad tick kill its timer or become an unhandled rejection. It fabricates nothing: it abandons the tick, reports via `services().diagnostics`, and retries next tick. One supervisor per loop; a nested one just hides the failure from the real one.
3. **"Carry on past a failure" requires naming the condition.** If carrying on is correct then the condition wasn't exceptional — so say which condition, and check it. `isHostUnavailable(error)` (from `ports/errors`) is that check: a host that isn't answering will not answer the next call either, so a caller that legitimately continues past a *refusal* (`patterns`/`generate`: not every element type accepts a link) must still rethrow when the board has simply stopped responding. A justification that reasons about one benign cause while the `catch` swallows all of them is a bug, not a decision.

Corollary: **never gate retry/heal state on a step that may have failed.** `redockSliceButton` used to record the new frame size even when the re-dock threw, so the size-change detector concluded the work was done and never retried — worse than silence.

**The Canvas contract:** creation ops (`createCard`/`createImage`/`createText`/`createContainer`/`createShape`/`createLink`/`createArrow`) each return a snapshot; queries (`get`, `getFresh` (bypasses the handle cache — for reads that act on current position/text, where stale is silently wrong; a dead id is omitted and evicted), `containers`, `childrenOf`, `connectors`, `connectorById` (one connector, fetched fresh; null = the board answered "no such connector"), `groups`, `selection`) return immutable `CanvasElement`/`CanvasConnector`/`CanvasGroup` snapshots; mutations are a batch of `ElementPatch`es via `apply()` plus structural ops (`addToContainer`, `group`/`groupMembers`, `remove`, `setConnectorColor`/`setConnectorCaption` (the only connector overrides, used only by the completeness flag — the caption replaces the line's text outright), `setMeta`/`getMeta`, `settle`, `setBulkMode` (a rate-limit pacing hint for bulk generation)). Coordinates are **local**: relative to the parent container if parented, absolute otherwise — the Miro adapter maps this 1:1 to the SDK's child-relative coords, and `addToContainer`/`settle` are the only bridges between the two spaces.

## SDK landmines (learned the hard way)

- The CDN `miro.js` is a bootstrap; the real SDK is whatever Miro currently ships. Runtime validation can be **ahead of `@mirohq/websdk-types`**: `createFrame` requires `style.fillColor` at runtime (types say optional) and rejects unknown props. When the runtime contradicts the types, trust the runtime error toast (`reportError` surfaces the real message).
- **Metadata and app data are invisible across separate app registrations.** Both `item.setMetadata`/`getMetadata` (the `em`/`fields-box` tags) and `board.getAppData`/`setAppData` (every `em-*` registry) are scoped per app — Miro's own docs: "An app can access only the metadata that it sets. It cannot access metadata set by other apps." Two Miro app registrations (two Client IDs — e.g. a permanent "dev" app on `localhost:3000` run alongside "prod", rather than one app whose App URL gets repointed per the README) installed on the same board can never see each other's tags or registries, even for the identical element: a screen's fields box tagged by one is invisible to the other, so its fields look unrecognized with no error and no fix from refreshing — reinstalling the *same* app doesn't change its Client ID, so whichever app created the data keeps reading it and the other never will. Not a bug to patch; full debugging story in DECISIONS.md.
- **App metadata is not universal.** `getMetadata`/`setMetadata` work on card, connector, embed, image, preview, shape, sticky note and text — and are **rejected at runtime** on frames, groups, and item types the SDK doesn't model ([reference](https://developers.miro.com/docs/websdk-reference-unsupported)). The published types declare the method on all of them, so nothing catches this at compile time and `typeof item.getMetadata` is no guard either — the call throws *"The specified command is unsupported: Frame.getMetadata()"*. `MiroCanvas.getMeta` keeps an allowlist and answers `null` for the rest without calling: an element that *cannot* hold a tag carries none, which is a true answer, and it's the named condition that lets the read carry on (selecting a slice threw once the blanket `catch` came out). `setMeta` is deliberately **not** guarded the same way — "it carries no tag" is true of a frame, but "I tagged it" never is.
- **Frames**: no nesting, no connector endpoints, no border styling (only `fillColor`), and children can survive frame deletion. Hence: board-app-data registries (`em-specs`, `em-slices`, `em-links`, plus `em-fields` for field-bearing elements, `em-flags` for reddened connectors, and `em-gen` for the generation checkpoint) instead of frame metadata, and explicit orphan cleanup. The registries also track frame width/height so polling can detect resizes.
- **Shrinking a frame evicts or deletes children** that fall outside the new bounds. App-placed frame chrome must be self-healing: verify the child still exists and is still parented before repositioning it, re-adopt or recreate otherwise (see `redockSliceButton`).
- **No SDK events** for item update, delete, resize, or click. Everything reactive is built from `selection:update` (a click IS a selection), the `drop` event, and polling — all surfaced through the `Runtime` port. Frame "buttons" are images with `em` metadata recognized on selection (`canvas.getMeta`).
- Registering listeners inside React effects caused duplicate items: StrictMode double-mounts, HMR re-evaluates modules, and `ui.off` is unreliable. Pattern (now in `MiroRuntime`): register the SDK listener once per page load behind a `window.__em*Registered` flag, with a `window.__em*Handler` indirection so HMR keeps logic fresh. **`selection:update` is the exception: it has many independent listeners** (the panel alone has three), so it keeps a `window.__emSelSubs` **Set** and `onSelectionChange` returns an unsubscribe. A single handler slot silently broke every listener but the last to register — on the Build tab, `ConvertSection`'s effect ran after `BuildingBlocksSection`'s and overwrote it, so the Screen tile never noticed selected images. Consequences: a caller that can go away (a React effect) **must** unsubscribe, and a caller that can't (the board script's module-scope subscribe in `src/index.ts`) needs a window flag, since adding stacks where replacing didn't.
- `selection:update` is a **push event and costs no API credits**, so the selection stream keeps working on a rate-limited board — but `getSelection()` is a real call and fails. Anything that seeds from it must not render the failure as "nothing is selected" (see `useSelection`).
- The `Unsupported` item type has `type: string`, so `item.type === 'frame'` narrowing fails on unions. The adapter normalizes every item's `type` to a `CanvasElement.kind`; use-cases narrow on `kind` (`'container'`, `'card'`, `'image'`, …), never on raw SDK types.
- Frame children use parent-top-left-relative coordinates. The adapter's convention: create at absolute coords, then `addToContainer` re-sets relative coords and `sync()`s. Items created over a frame may get auto-captured with shifted coords — `MiroCanvas.settle` (the `canvas.settle` port op) re-pins them; a non-capturing canvas no-ops it.
- Sticky `fillColor` is a fixed palette; the four model colors (`orange`, `blue`, `light_green`, `yellow`, plus `red` for errors) are native values. Shape text does NOT scale on resize — scalable graphics are inline SVGs shipped as base64 data URLs (`createImage`).
- Every awaited SDK call is a round-trip (~tens of ms). Batch independent creations with `Promise.all` (spec creation does this); sequential chains of 20+ calls feel slow to the user. **But** Miro rate-limits writes (HTTP 429): a single user action is fine, but a *bulk* op that bursts dozens of creates trips it. So `MiroCanvas` wraps every write (creates, `sync`, `add`, `setMetadata`) in `withRetry` backoff, and bulk callers like `generate` create **sequentially** instead of `Promise.all`. Per-action features keep using `Promise.all`.
- **Miro enforces two credit budgets and they fail differently.** `100000 credits per minute` is a burst ceiling — transient, and retrying rides it out. `1000000 credits per hour` is a *sustained* ceiling (~16.7k/min averaged): you can sit far under the burst ceiling and still exhaust it over an hour, after which nothing works until the window rolls. Retrying the hourly one only spends more of a budget that is already gone, so `rateLimit.ts` tells them apart by the message ("per hour" / "per minute"), fails the hourly case fast, and sets a cooldown that every background loop honours. **Usage is reported, but not to the Web SDK**: the `X-RateLimit-Remaining` headers are on *REST* responses, and the SDK hands back resolved promises, never an HTTP response — so the Console's credit meter *counts* our own calls rather than reading a balance. It is labelled "spent by this app" for that reason, and the 429 stays the only ground truth (and the only thing that throttles anything). Limits are **per user per application**, so other Miro apps don't share this budget. See `domain/credits` and DECISIONS.md.
- **`miro.board.get` costs 500 credits — Weight Level 3**, the same as creating an image, and 10× most of what we call. **Don't infer a weight: every element page in the [SDK reference](https://developers.miro.com/docs/sdk-reference) prints the level under each method** ("🚦 Rate limit: Level 3"), and `adapters/miro/weights.ts` is where they're recorded. Level 3 (500) covers `board.get`, `board.getSelection`, `frame.getChildren`, `item.getConnectors`, and **creating *or updating*** an image or embed — so `image.sync` is 500 too, which matters because screens/automations *are* images. Level 1 (50) covers every other create, `getMetadata`, `getAppData`, `getById`, `group`/`ungroup`/`remove`/`deselect`/`getInfo`. The docs' one gap: **write methods print no level** (`sync`, `setMetadata`, `setAppData`, `frame.add`) — those are inferred Level 1. So **the number of Level 3 calls essentially *is* the cost**: a pass making three `board.get`s costs ~1,550 credits, and at a fixed 4s that alone is ~1.4M/hour against a 1M budget — over, on an empty board, before anyone does any work. Count them per hour when touching anything that runs unattended, and batch ids into one call rather than fetching per group (see `completeness.analyze`). The reverse holds for one or two **known** ids: `board.getById` is Level 1 ×n, so `MiroCanvas.getFresh`/`connectorById` beat one `board.get({id})` up to ~10 ids — and getById names a dead id outright ("Can not retrieve item with id …"), which is what lets `getFresh` evict stale cache entries.
- **Board app data has a tight total budget** — writes failed at a measured ~31 KB across all `em-*` keys. Keep registries lean: no per-field ids, no sticky field records (parsed from the sticky's text on demand), and the `em-gen` checkpoint auto-expires after 24h. Size-measuring snippet and breakdown in [docs/DECISIONS.md](docs/DECISIONS.md).
- Failed `eventhub.eu01.miro.com` requests in devtools are Miro telemetry blocked by ad-blockers/VPNs — **not** the app's rate limit; don't diagnose from them. The real signal is a REST 429 from `api.miro.com` (details in [docs/DECISIONS.md](docs/DECISIONS.md)).

## Product decisions (do not regress)

These came from explicit user feedback; fuller context (what was tried and rejected, and why) in [docs/DECISIONS.md](docs/DECISIONS.md):

- Never auto-zoom on creation. `ensureVisible` only expands the viewport, never zooms in.
- Connectors use SDK defaults — zero shape/style overrides. No arrow-styling features. The completeness flag is the one exception: it reddens an under-supplied arrow and captions it with the missing fields.
- Linking items is manual (Miro's own connector tool); a chain-mode feature was built and removed. Pattern stamps may pre-link their own items.
- Lean panel: no app-name header, no close button, no redundant controls. On-board affordances (the + buttons) are preferred over panel buttons for board-targeted actions.
- Screens and automations are grouped title-text + image pairs — not frames (connectors), not shapes (text scaling, accidental text-edit). Slices and specs are frames (containment is the point).
- Don't decorate frames with child shapes (a slice border attempt hid the frame title).
- Chapters are single blue connectors (`#61DEFF`, caption above the line) placed above the model to group slices into contexts; sub-chapters are optional narrower arrows beneath a chapter — many chapters stand alone.
- Spec copies are **typed, editable blocks**, not frozen pictures: a copy carries its source's type (so the Properties editor recognizes it and it can hold its own fields), keeps a navigable back-link to the original, and is the user's to edit. Only **color** propagates original→copy — never label, text, or fields — and never the reverse (`syncSpecCopies`).
- Fields: errors can't carry fields (product decision). Cards embed their fields in their own text (first line = the name, so a native rename just edits it); screens/automations get an attached box beneath them, grouped so it travels with the element. **Stickies keep no registry record** — their fields live in the sticky's own text and are parsed from it on demand (the panel editor and the completeness check both read the board, never a stored copy). Only **box-mode** screens/automations keep an `em-fields` record, because their fields render into a *separate* box shape that a frame-shrink can evict or delete, and the record is what lets housekeeping rebuild it — but the record is a **follower, not the source of truth**: the box's text is user-editable like a sticky's, and manual box edits are adopted into the registry (panel poll + housekeeping), never overwritten. Housekeeping fits a box's size/position to the lines it shows but never rewrites its text (content is written only when rebuilding a missing box). Emptying the box clears the fields like on a sticky, and housekeeping then evicts the empty box and drops its record (same end state as clearing fields from the panel); a field-less record whose box is already gone is pruned, not rebuilt. The box carries a `fields-box` metadata tag, and **only a tagged shape is ever treated as a fields box** (recovery scan, completeness) — a user-drawn shape grouped with the same element must never be hijacked; housekeeping stamps the tag onto pre-tag boxes via their registry record. The completeness check reads fields straight from the notes/boxes, so it never depends on the registry; arrows recolor on a ~4s headless poll, nudged immediately after a panel field edit.
- **Field type plays no part in completeness matching — only the name does.** A required `email : string` downstream is satisfied by an upstream field named `email` regardless of what type either side declares; `fieldMatchKey` is just the field's name. Type still renders everywhere a field is shown (`formatField` always writes `name : type`) — it's just not part of what makes two fields "the same field" for the check. Applies uniformly: the plain-name match, the alias match, and the zero-contribution check below all go through this same name-only key.
- Optionality is read on **both sides** of the completeness check: an optional field need not be supplied by anyone, *and* supplies nobody — a field that may be absent can't guarantee a required one, so `email : string?` upstream does not satisfy a required `email : string` downstream. A required supply from any one source in the fan-in still satisfies the target. The caption distinguishes the two gaps: a field nobody carries at all is the bare key (`phone : string`), while a field carried only as optional reads `Field "email : string" is required` (the full `name : type` display in the quotes, matching how the field is shown everywhere else — type plays no part in the match itself, per the bullet above, but still renders for documentation). Both wordings can share one arrow; uniform wording was declined.
- **A zero-contribution link reddens even when its target is fine.** The per-target pooling above is deliberately loose — an arrow whose source supplies nothing is silent as long as its siblings cover the target — but that let a link that's wired up yet carries none of the target's fields hide behind a healthy pool. So once a target *is* covered, each arrow into it is checked on its own: a source supplying zero of the target's required fields (an optional-only match still counts as zero) is reddened and captioned `Supplies none of the required fields`, distinct from the missing-field wording. Supplying just one relevant field clears the bar, so a source carrying its own slice of a multi-event hydration is never caught by this — only an arrow contributing nothing at all is.
- A field can name the differently-named upstream field(s) that feed it — a read model's `name` supplied by an event's `full_name` renders **`full_name > name : string`** on the block and satisfies the check. **Source first, arrow pointing downstream**, so the line scans the way the canvas does. The alias is **intake-only** (`fieldMatchKey` ignores it, so the read model still supplies `name : string` downstream — aliases never propagate); type plays no part in an alias match either, same as the plain-name match (see the type-matching bullet above) — `b > a` is satisfied by an upstream `b` of any type. It's **per target, not per arrow** (any source in the fan-in carrying `b : string` satisfies it). **More than one upstream name is comma-separated** — `full_name, display_name > name : string` — for a read model whose `name` might arrive as either, from different events; any one name satisfies it, they're alternatives for the same datum, not all required from the same arrow. This also feeds the zero-contribution check above: a source counts as contributing the moment it matches *any* declared name, so a read model hydrated by one event carrying `full_name` and another carrying `display_name` has both arrows clean rather than one reddening as a false zero-contribution. It lives in the block's own text like every other field, never a registry, so the check reads it off the board for free and it stays visible. `>` therefore joins `:` as a character a field name can't contain (stripped in `cleanFieldName`, both panel inputs, and the generator's trust boundary) — `,` is deliberately left alone, since it's the separator between upstream names, not a name character. **ASCII, not a glyph** — the sticky's text is the store and is edited on the board, so the rendered form is the form people have to type; nothing parses `←`, `<-` or `<` (none was ever deployed, so there's no legacy to carry). The panel's toggle button still *shows* `→`, since it's clicked rather than typed. `>` is escaped to `&gt;` on write, and is the safer bracket: `stripHtml`'s tag regex only starts matching at a `<`, so a stray `>` can't be read as the tail of a tag. The caption names the field as the target displays it (`full_name, display_name > name : string`), because the arrow says which *upstream* field(s) are missing. The generator never emits aliases.
- **A field can be marked `generated`** — synthesized by the block itself at runtime (a command handler assigning an id, say) rather than sourced from an incoming block. Renders as `id : UUID!` (the `!` sits right after the type, before the optional `?` if both apply — rare, but `id : UUID!?` parses). **Excluded from what its own block's incoming arrows must supply** — `requiredFieldsById` drops it, so nothing upstream is ever expected to carry it and no arrow reddens for its absence — **but still counts as a guaranteed supply to whatever the block feeds downstream**, exactly like any other required field (`suppliedFieldsById` keeps it) — once generated, it's as real and present to a consumer as one that arrived from upstream. This is the opposite asymmetry from `optional` (which is weak on *both* sides): `generated` is only weak on the *intake* side. Panel has a `!` toggle button next to the `?` one. The generator doesn't emit it, same as aliases.
- **A field can be marked `collection`** — many of the declared type rather than one, purely a display detail with no bearing on the completeness check (which matches by name only, per the type-matching bullet above). Renders as `[]` right after the type label, closest to the type since it modifies the type itself rather than the field as a whole — before the generated `!` and optional `?` marks if either apply: `tags : string[]`, or, rarer, `ids : UUID[]!?`. Parsed outside-in like the other two marks (`?` first, then `!`, then `[]`), so it composes with both without new cases. Panel has a `[]` toggle button next to the `!` one. The generator doesn't emit it, same as `generated` and aliases.
- **A field can carry an `example`** — a sample value, free text, written last on the line after every mark: `name : string = Ada Lovelace` (full order `from > name : type[]!? = example`). Pure documentation: no part in the completeness check (matching stays name-only) and **never in an arrow caption** — a caption reports a *missing* field, and a sample of a value nobody carries would read as data. Parsed before the marks (it's the outermost part): everything past the first `=` on the type-label side is the example, verbatim — commas, colons, further `=`s all land in it — so it composes with `?`/`!`/`[]` without new cases and an example ending in one of those stays intact. `=` lives only past the colon, so names may contain it (`cleanFieldName` unchanged); the one casualty is a custom type whose own name contains `=` — its tail reads back as an example — the same class of tradeoff already accepted for the marks. Panel: an `=` toggle after `→` with the alias toggle's exact on/off contract (on = reveal the input locally, off with text = a real edit), the input stacked under the fed-by input; the collapsed accordion row shows the example on a **second, dimmer line** (the one part of the notation that leaves the line — free text is arbitrarily long, and inline it would truncate the notation itself away; chosen over inline-dimmed and a right-aligned pill). The generator doesn't emit it, same as the marks and aliases.
- **Selecting a single attached arrow turns the Properties tab into the arrow toolset** (`ArrowSection` / `features/arrow`): **Copy →/← merges** (name-keyed, the receiver's version of a shared name is never touched), **Replace →/← overwrites** the whole list, and a **Navigate to \<block\>** button per attached end pans the viewport (`panTo`: current zoom kept, grow only to fit — the navigate counterpart of never-zoom-in). A transferred field keeps type/`[]`/`?`/example and **drops the alias and `!`** (both describe the block they were written on, not the receiver). Transfers from an empty source are refused — replace-with-nothing is a disguised clear. Everything acts on **fresh reads at click time** (`getFresh`/`connectorById`, getById at Level 1 — never the stale handle cache) and stores nothing: no registry, no app-data, writes ride the fields edit path (conflict check, box rebuild, completeness nudge), with the fresh read noted as the conflict base so it compares now-against-now. Rationale in DECISIONS.md.
- **The Properties tab (renamed from Fields) opens with the element's name, editable** (`NameEditor` / `features/naming`): the name lives where Miro keeps it — a sticky's first text line (`replaceFirstLine` rewrites only that line, so a note's prose or a block's field lines are never round-tripped through the field parser), a screen/automation's grouped title text (recreated by the next rename if deleted), a slice/spec/plain frame's own title (the `title` `ElementPatch`), a swimlane shape's own text, a chapter's caption (`setConnectorCaption`; a **free-floating** connector is a chapter and gets the name editor, an attached one gets the arrow toolset). No registry — resolution and rename both read fresh (`getFresh`/`connectorById`) and write straight to the board. Field-less named elements (slices, specs, swimlanes, chapters, notes, errors) get the name editor alone; the auto-jump to the tab stays limited to blocks and connectors, so selecting a frame to move it never yanks the panel.
